# Student Accounts Design

## Goal

Add optional student accounts so students can preserve their submission history across browsers and devices. Anonymous usage remains fully supported and is the fallback for students who skip registration.

---

## Key Decisions

- **Account fields:** username, email, password (min 8 chars). No email verification. No password reset (no email-sending capability).
- **Anonymous history:** fresh start on account creation — existing anonymous attempts are not linked to the new account.
- **Analytics:** unchanged — instructors still see session IDs, not usernames. Accounts are purely for the student's benefit.
- **Landing page flow:** auth-first, two-step on the same page (`/student`). Step 1 shows login/register. Step 2 (after auth or "continue anonymously") shows the class code form.
- **Session window:** 30-day sliding, stored in an httponly `student_session_token` cookie.

---

## Architecture

### New DB Tables (in `db.py`)

All three tables are created with `CREATE TABLE IF NOT EXISTS` in `init_db()`. No changes to existing tables.

```sql
CREATE TABLE IF NOT EXISTS student_users (
    id TEXT PRIMARY KEY,
    username TEXT UNIQUE NOT NULL,
    email TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS student_sessions (
    token TEXT PRIMARY KEY,
    student_id TEXT NOT NULL REFERENCES student_users(id) ON DELETE CASCADE,
    expires_at TIMESTAMP NOT NULL
);

CREATE TABLE IF NOT EXISTS student_question_sessions (
    id TEXT PRIMARY KEY,
    student_id TEXT NOT NULL REFERENCES student_users(id) ON DELETE CASCADE,
    question_id TEXT NOT NULL REFERENCES questions(id) ON DELETE CASCADE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(student_id, question_id)
);
```

`student_question_sessions.id` is a UUID that becomes the `session_id` stored in the `attempts` table — the glue between accounts and the existing analytics model. The `UNIQUE(student_id, question_id)` constraint will be relaxed when "multiple sessions" is implemented.

### New `db.py` Functions

```python
create_student_user(username, email, password_hash) -> str          # returns id
get_student_by_username(username) -> dict | None
get_student_by_email(email) -> dict | None
get_student_by_id(student_id) -> dict | None
create_student_session(token, student_id, expires_at) -> None
get_student_session(token) -> dict | None                           # returns row only if expires_at > datetime('now') (SQL-level check)
update_student_session_expiry(token, expires_at) -> None
delete_student_session(token) -> None
get_or_create_question_session(student_id, question_id) -> str      # returns session UUID
```

`get_or_create_question_session` uses `INSERT OR IGNORE` followed by `SELECT id` to be idempotent and race-safe.

All new `db.py` functions must use `_connect()` (the private helper already used by every existing function in `db.py`) rather than calling `sqlite3.connect()` directly — `_connect()` enables `PRAGMA foreign_keys = ON` and sets the row factory.

`get_student_session` filters with `WHERE token = ? AND expires_at > datetime('now')` so expired sessions are treated as missing at the DB layer.

**Token generation:** `create_student_session` is called with a token generated by `generate_token()` — the same helper used for instructor sessions. Both register and login routes call `generate_token()` to produce the cookie value before calling `create_student_session`.

**Namespace note:** Student usernames and instructor usernames live in separate tables (`student_users` vs `users`). There is no conflict if both have the same username.

### New Routes (in `app.py`)

All five routes are registered before `# ---- Auth API routes ----`. Add all nine new `db.py` functions to the `from db import (...)` block at the top of `app.py`. `hash_password`, `verify_password`, and `generate_token` are already imported from `auth` in the existing `from auth import (...)` block and require no change.

#### `POST /api/student/auth/register`

Request body: `StudentRegisterRequest(BaseModel)` with fields `username: str`, `email: str`, `password: str`.

- Raise `HTTPException(status_code=400)` if password < 8 chars
- Raise `HTTPException(status_code=400)` if username already taken (check `get_student_by_username`)
- Raise `HTTPException(status_code=400)` if email already taken (check `get_student_by_email`)
- No email format validation — email is used only for login lookup, not for sending messages (no email-sending capability)
- Hash password with `hash_password` (same bcrypt helper used for instructors)
- Call `generate_token()` to produce the session token value
- Call `create_student_user` first (to obtain `student_id`), then call `create_student_session(token, student_id, expires_at)` — this order is required because the session row references the user id via a foreign key. `expires_at` is `(datetime.now(timezone.utc).replace(tzinfo=None) + timedelta(days=30)).strftime("%Y-%m-%d %H:%M:%S")`.
- Set `student_session_token` cookie (httponly, samesite=lax, max_age=30×24×3600)
- Return `{ id, username }`

#### `POST /api/student/auth/login`

Request body: `StudentLoginRequest(BaseModel)` with fields `username_or_email: str`, `password: str`.

- Look up by username first (`get_student_by_username`), then by email (`get_student_by_email`) if not found
- Raise `HTTPException(status_code=401)` if not found or password mismatch (`verify_password`)
- Do not invalidate existing sessions — students may be logged in on multiple devices simultaneously (no `delete_sessions_for_user` call)
- Call `generate_token()`, create session, set cookie (same as register)
- Return `{ id, username }`

#### `POST /api/student/auth/logout`

Signature: `student_session_token: str | None = Cookie(default=None)`

- Call `delete_student_session(student_session_token)` if token is present (no-op if token is missing or already expired)
- Use `response.delete_cookie("student_session_token", httponly=True, samesite="lax")` to clear the cookie — this sets `Max-Age=0` with the correct `path="/"` default so browsers honour the deletion (do not use `response.set_cookie(value="", max_age=0)` without `path`, as it may fail to clear cookies that were set with `path="/"`)
- Return `{ ok: true }`

#### `GET /api/student/auth/me`

Signature: `student_session_token: str | None = Cookie(default=None)`

- Pass token to `_validate_student_session` (checks expiry, slides window with `update_student_session_expiry`)
- Raise `HTTPException(status_code=401)` if missing or invalid
- Return `{ id, username }`

#### `GET /api/student/session/{question_id}`

Signature: `student_session_token: str | None = Cookie(default=None)`

- Pass token to `_validate_student_session` (same as `/me`, including sliding the 30-day window); raise `HTTPException(status_code=401)` if invalid
- `get_question(question_id)` → 404 if not found
- No class membership check — any authenticated student can get a session for any question (consistent with how anonymous students access questions by class code today)
- `get_or_create_question_session(student_id, question_id)` → returns UUID
- Return `{ session_id: UUID }`

### Student session validation helper

A module-level helper added directly to `app.py` (note: the analogous `_validate_session` actually lives in `dependencies.py` and is imported into `app.py`, but `_validate_student_session` is simpler and can live directly in `app.py` since it is only called from student routes):

```python
def _validate_student_session(student_session_token: str | None) -> dict | None:
    if not student_session_token:
        return None
    session = get_student_session(student_session_token)
    if not session:
        return None
    new_expiry = (datetime.now(timezone.utc).replace(tzinfo=None) + timedelta(days=30)).strftime("%Y-%m-%d %H:%M:%S")
    update_student_session_expiry(student_session_token, new_expiry)
    return get_student_by_id(session["student_id"])
```

**Concurrent tab note:** If two tabs call `/me` simultaneously, both will call `update_student_session_expiry` with the same new expiry. This is safe — the last write wins and both produce the same result. No locking needed.

## UI Changes

### Landing page (`templates/student.html`, `static/app.js`)

**New Step 1 — auth panel** (shown when `mode == "landing"` and student is not logged in):

```
Sign in or create account
[Username or email         ]
[Password                  ]
[        Sign in           ]
[      Create account      ]

No account? Continue anonymously — then enter your class code on the next panel.
```

**Step 2 — class code panel** (shown after successful auth, or after "continue anonymously"):

If logged in:
```
Already signed in as alice · Sign out

Enter your class code
[BIO101    ] [ Continue → ]
```

If anonymous (skipped auth):
```
Browsing anonymously · Sign in

Enter your class code
[BIO101    ] [ Continue → ]
```

**Implementation:** `initStudentLanding()` in `app.js` calls `GET /api/student/auth/me` on load. If 200 → skip the auth panel, go directly to Step 2 with the logged-in state. If 401 → show the auth panel (Step 1). After successful register/login or clicking "Continue anonymously," transition to Step 2 (hide Step 1, show Step 2) without a page reload.

The existing `essay_coach_class_id` localStorage check (auto-redirect to question list) is preserved — it runs after the auth check, so a logged-in student with a saved class_id goes straight to the question list.

### Workspace (`static/app.js`)

**Session resolution** — `initStudent()` calls `GET /api/student/session/{QUESTION_ID}` before loading attempt history. If 200, use the returned `session_id`. If 401 (anonymous), fall back to `getSessionId()` (localStorage). The resolved session_id is stored in a module-level variable and used for all subsequent API calls in that page load.

**Identity indicator** — a small line above the question prompt (rendered by JS after session resolution):
- Logged in: `Already signed in as alice · Sign out`
- Anonymous: *(nothing — no indicator in the workspace for anonymous users)*

"Sign out" calls `POST /api/student/auth/logout` then reloads the page (returning to Step 1 of the landing page).

## Testing (`tests/test_student_auth.py`)

17 integration tests using the same `client` fixture and `fresh_db` autouse fixture pattern as `tests/test_analytics_integration.py`. The `fresh_db` fixture must patch both `config.DATABASE_PATH` and `db.DATABASE_PATH` (exactly as in `test_analytics_integration.py`):

```python
@pytest.fixture(autouse=True)
def fresh_db(tmp_path, monkeypatch):
    db_path = str(tmp_path / "test.db")
    monkeypatch.setattr("config.DATABASE_PATH", db_path)
    monkeypatch.setattr("db.DATABASE_PATH", db_path)
    db.init_db()
    yield
```

Tests that need to inspect raw DB rows should open `sqlite3.connect(db_path)` directly using the `tmp_path` fixture value.

**Register (4 tests):**
1. Success → 200, `student_session_token` cookie set, returns `{ id, username }`
2. Duplicate username → 400
3. Duplicate email → 400
4. Password under 8 chars → 400

**Login (4 tests):**
5. Success by username → 200, cookie set
6. Success by email → 200, cookie set
7. Wrong password → 401
8. Unknown user → 401

**Me (2 tests):**
9. Authenticated → 200, returns `{ id, username }`
10. Unauthenticated → 401

**Logout (1 test):**
11. Clears session — subsequent `GET /api/student/auth/me` returns 401

**Session endpoint (4 tests):**
12. Authenticated, question exists → 200, returns UUID
13. Authenticated, same question second call → same UUID (idempotent)
14. Authenticated, question not found → 404
15. Unauthenticated → 401

**Session expiry and sliding window (2 tests):**
16. Expired session token → `GET /api/student/auth/me` returns 401 and does not slide the window. To manufacture an expired session: call `create_student_session(token, student_id, "2000-01-01 00:00:00")` directly (bypassing the normal route). Then call `/me` with that token and verify the 401 response. To confirm `expires_at` was not updated, open a raw `sqlite3.connect(db_path)` connection (using the same `tmp_path`-based path injected by the `fresh_db` fixture) and `SELECT expires_at FROM student_sessions WHERE token = ?` — assert it is still `"2000-01-01 00:00:00"`.
17. Valid session sliding window — query `expires_at` from DB before calling `GET /api/student/auth/me`, then call `/me` and query `expires_at` again; assert the new value is strictly greater than the original (use the same direct `sqlite3.connect(db_path)` approach as test 16).
